<objective>
Proactive recurring renewal without a broker (D-08, Celery/RQ/Arq banned): an in-process
APScheduler `AsyncIOScheduler` daily sweep, started in the FastAPI `lifespan`, that finds active
subscriptions whose `current_period_end` is within a grace window and issues a merchant-initiated
charge via `PaymentService.renew_subscription` (Plan 03). The DB window stays the source of truth —
the lazy consume-gate backbone (Plan 04) already enforces entitlement, so the sweep is the
proactive layer, and a missed tick self-heals because each run recomputes "who's due" from the DB.

Purpose: ЮKassa does NOT auto-charge — the merchant must trigger each renewal. This is the only new
moving part the phase needs; keeping the trigger in `core/scheduler.py` (one call site) makes the
documented timeweb-cron fallback a swap if the deploy ever scales beyond one instance (A2).
Output: `core/scheduler.py` + lifespan wiring + a `find_due_subscriptions` sweep query + a unit
test; `APScheduler` added behind a legitimacy checkpoint.
</objective>
