synchronous official `yookassa` SDK via `anyio.to_thread.run_sync`, owning create-payment (no

`SUBSCRIPTION_WINDOW_UNLIMITED` sentinel and the exact `subscription_spreads_limit`/`_used`
semantics the grant + renewal write, which Plan 04's consume-gate then reads.

Purpose: This is where the security invariants live — grant ONLY on the re-fetched `succeeded`,
exactly-once via the conditional `UPDATE ... WHERE status=CREATED ... RETURNING` + UNIQUE
`provider_payment_id`, server-recomputed price, no event-loop blocking. Concentrating them in one
service keeps the router (Plan 04) thin and the gate (Plan 04) consuming a single, pinned contract.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ЮKassa webhook → service | Unsigned; the service NEVER trusts the body status — it re-fetches by id and trusts only the API-confirmed status. |
| service → ЮKassa API | Authenticated (HTTP Basic via SDK config); secret only in env; every call off the event loop. |
| service → DB (grant) | The conditional status-transition UPDATE + UNIQUE provider_payment_id is the exactly-once boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-WEBHOOK-FORGE | Spoofing | grant decision | mitigate | Grant only after `find_payment` re-fetch returns `succeeded`; the body status is never read for granting (Pattern 3). |
| T-07-REPLAY | Tampering/Elevation | repeated `payment.succeeded` | mitigate | Conditional `UPDATE ... WHERE status=CREATED ... RETURNING`; `.first() is None` ⇒ no double grant; UNIQUE `provider_payment_id` backstop (Pattern 5). |
| T-07-GRANT-ON-CREATE | Elevation | create-payment | mitigate | `create_payment` mutates no balance; only the grant path (post re-fetch) does (Pattern 2; asserted by the no-grant test). |
| T-07-AMOUNT | Tampering | charge amount | mitigate | Amount always recomputed from `product.stars_price` via `format_rub`; never a client value. |
| T-07-REFUND-OVERCREDIT | Tampering | refund access adjustment | mitigate | Refund decrements `paid_spreads_balance` by `Product.spreads_amount` (the granted entitlement), clamped `>= 0` — never by the RUB `payment.amount`. |
| T-07-DOUBLE-CHARGE | Tampering | recurring renewal | mitigate | Deterministic `renew:<sub_id>:<period_index>` Idempotence-Key (24h ЮKassa idempotency); retry reuses the same key (Pitfall 5). |
| T-07-OPEN-REDIRECT | Tampering | `return_url` | mitigate | `return_url` is server-constructed (app deep link), never client-supplied. |
| T-07-SECRET-LEAK | Info Disclosure | shop secret / `raw_update` | mitigate | Secret only from `settings` (env); `raw_update` JSONB is server-side audit, never returned by this service. |
| T-07-LOOP-BLOCK | DoS | sync SDK in async path | mitigate | Every SDK call via `anyio.to_thread.run_sync` (Pitfall 3). |
</threat_model>
