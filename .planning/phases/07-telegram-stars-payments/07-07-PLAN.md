<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → create/cancel API | Bearer JWT via apiFetch; the server scopes to the JWT identity (no client identity sent). |
| Telegram WebView → external bank page | `openLink` opens the ЮKassa-hosted page; the app never sees card data (PCI stays with ЮKassa). |
| client trust in grant | The client NEVER self-grants; it only polls `GET /api/me` for the server-confirmed state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-CARD-DATA | Info Disclosure | payment page | mitigate | `openLink(confirmation_url)` hands off to the ЮKassa-hosted form; the app never touches PAN. |
| T-07-CLIENT-GRANT | Elevation | shop UI | mitigate | The UI grants nothing — it polls `GET /api/me` for the webhook-confirmed balance/sub (server is truth). |
| T-07-AMOUNT | Tampering | buy request | mitigate | The buy call sends only `product_slug`; price is server-recomputed (no amount field client-side). |
| SAFE-06 | (brand) | shop copy | mitigate | All shop/success/failure strings are copy.ts constants scanned by the brand-token test; honest «деньги не списаны» on failure (D-13). |
| T-07-OPEN-REDIRECT | Tampering | openLink target | mitigate | The URL opened is the server-returned `confirmation_url` (built server-side from the app deep link), never a client-constructed URL. |
</threat_model>
