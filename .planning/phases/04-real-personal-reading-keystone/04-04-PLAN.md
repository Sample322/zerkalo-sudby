<objective>
Build the PromptEngine that turns a drawn hand + deck/spread/safety context into ONE fused `messages.parse` prompt (the §17 single-card task and the §18 summary task merged into one call producing the whole `ReadingOutput`), and author the content the engine needs: the mandatory D-02 per-deck signature instruction for all 6 decks, the generic D-04 refusal copy, and the D-06 abusive-redirect copy — seeded into `prompt_templates`. This is where the Core Value ("same question, different deck") is engineered: tone + focus from the §19 deck_modifier, a guaranteed signature per deck, uniform length/structure (D-10/D-11), always Russian (D-14).

Purpose: The engine is the bridge between the locked, already-seeded §16/§17/§18/§19 templates and the single structured call — it composes them, injects live card data, conditionally appends the safety_modifier, and resolves the versioned `prompt_version` for the audit trail.
Output: `services/prompt_engine.py`, an updated `prompts.json` (D-02 signatures + generic refusal + redirect copy), and `tests/unit/test_prompt_engine.py`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user question + card data → assembled prompt | The question is interpolated as DATA inside the fixed §16 system frame; it never becomes an instruction. |
| prompt templates (DB) → prompt | Versioned templates are the admin-controlled safety valve (Phase 8); the engine must read the ACTIVE versions, not hardcode copy. |
| generated copy → user | The system frame's allow/ban formulations (§15.1) + the safety modifier shape the output to avoid categorical/fatalistic phrasing (SAFE-04/05). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-19 | Tampering / Elevation | Prompt injection via the question overriding the system frame | mitigate | Question interpolated as labelled data inside the §16 fixed frame; the model is bound to the `ReadingOutput` schema (Plan 03); no instruction-following from user text. |
| T-04-20 | (Safety) | Output uses categorical/fatalistic phrasing or asserts another's feelings as fact | mitigate | System §16 + §15.1 ban-list baked into the assembled prompt; the safety modifier (§20.3) appended for sensitive topics (D-05); the brand/soft-phrasing guard (Plan 02) backstops output (SAFE-04/05). |
| T-04-21 | (Safety) | Crisis/abusive question handled by generation instead of refusal/redirect | mitigate | The refusal (generic, D-04) + redirect (D-06) copy is authored here; Plan 05 returns it before any draw/generation when the safety route says so. |
| T-04-22 | Repudiation / audit | Unversioned prompt makes a bad generation untraceable | mitigate | `prompt_version` composed from active template versions → persisted to readings + generation_logs (ANALYTICS-02); admin version toggle is the Phase-8 valve. |
</threat_model>
