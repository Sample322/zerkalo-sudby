<objective>
Product analytics into the already-existing `app_events` table (`models/analytics.py`), which has
ZERO writers today. Pure add, split by origin (08-RESEARCH.md §2):

- **Server-side** reliable events emitted where they truly happen — `reading_started/completed/
  failed` in the reading service, `payment_succeeded`/`subscription_started` in the payment grant
  path — via a best-effort `record_event` helper that opens its OWN short-lived session and commits
  independently, so a failed analytics INSERT can never abort the reading/payment transaction.
- **Client-side** funnel events via a thin `POST /api/events` (Bearer, JWT-scoped user_id, allowlist-
  validated, always 2xx) + a fire-and-forget FE `track()`.

The HARD invariant (same lesson as the P6 throttle fail-open): analytics is best-effort and must
NEVER slow, block, or break the core reading/payment/UI flow.
</objective>
