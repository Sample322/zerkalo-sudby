<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram WebView → POST /api/auth/telegram | The single most security-critical boundary: untrusted `init_data` crosses here; the entire identity model rests on validating it. |
| client → Bearer-protected routes | JWT echoed on every later call; forged/expired tokens must be rejected. |
| caller → admin routes | Privilege boundary: only `telegram_id ∈ ADMIN_TELEGRAM_IDS` may pass `require_admin`. |
| any handler → client (errors) | Internal failures must not leak stack traces / internals across this boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | Spoofed `telegram_id` via `initDataUnsafe` / request body | mitigate | Server-side two-stage HMAC; `telegram_id` derived ONLY from validated `user` (Pattern 3); unit + integration tested with forged/tampered inputs. |
| T-04-02 | Tampering/Spoofing | Replay of a leaked `initData` string | mitigate | `auth_date` freshness window (`INITDATA_MAX_AGE_SECONDS`, default 24h); stale → 401; tested. |
| T-04-03 | Tampering | Forged/tampered JWT or `alg:none` bypass | mitigate | PyJWT HS256 with `JWT_SECRET`; `decode` pins `algorithms=["HS256"]` (rejects `none`); signature/exp verified; tested. |
| T-04-04 | Elevation of Privilege | Non-admin reaching admin endpoints | mitigate | Server-side `require_admin` allowlist check; frontend guard cosmetic; `GET /api/admin/ping` 403/200 tested. |
| T-04-05 | Information Disclosure | Timing attack on hash/token compare | mitigate | `hmac.compare_digest` (constant-time) for the HMAC compare. |
| T-04-06 | Information Disclosure | Stack-trace / internal-error leakage to client | mitigate | Global exception handler → soft in-character JSON; details logged server-side only (INFRA-05). |
| T-04-07 | Information Disclosure | Auth error reveals which check failed (oracle) | mitigate | Endpoint returns a single generic 401 message regardless of failure cause. |
| T-04-08 | Information Disclosure | Secret (`JWT_SECRET`/`BOT_TOKEN`) leakage in logs/bodies | mitigate | Secrets only from env; never logged; never in error bodies (V8). |
| T-04-09 | Tampering | SQL injection in user upsert | mitigate | SQLAlchemy parameterized `insert(...).on_conflict_do_update` — no string-built SQL. |
</threat_model>
