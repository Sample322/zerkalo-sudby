<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram WebView → frontend JS | The browser holds the only real `initData`; the frontend forwards it unmodified to the backend (it does NOT trust/parse identity itself). |
| frontend → backend API | JWT stored client-side is echoed as Bearer; the backend remains the authority (frontend guards are cosmetic). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | Frontend trusting `initDataUnsafe` for identity | mitigate | Frontend forwards raw `initData` only; identity is derived server-side (Plan 04). No client-side trust of `user.id`. |
| T-05-02 | Information Disclosure | Stack trace / internals shown in the WebView on auth failure | mitigate | `AuthGate` renders an in-character error state; no error internals surfaced; brand-voice clean. |
| T-05-03 | Tampering | JWT exfiltration via XSS in the WebView | accept | MVP stores JWT in-memory (Zustand), short-lived; WebView CSP `connect-src` restricted to the API origin (Open Question #5). Hardening (storage strategy, full CSP) revisited at the deploy/polish phase. |
| T-05-04 | Spoofing | API origin not reachable due to CSP `connect-src` misconfig (also a functional risk) | mitigate | `VITE_API_BASE` env-driven; checkpoint step 3 verifies the WebView can reach the API; documented requirement. |
</threat_model>
